---
description: 코드 수정 시 안전 수칙 및 백업 워크플로우 (Safeguard Rules)
---

이 프로젝트의 코드를 수정하거나 기능을 업데이트할 때는 기존 로직의 유실을 막기 위해 **반드시** 다음 단계를 순서대로 엄격히 준수해야 합니다.

1. **상태 백업 (Backup)**
   - 코드를 수정하기 전에, 수정이 예정된 주요 파일들을 `_archive/YYYYMMDD_HHMM/` 디렉토리에 복사하여 백업본을 생성하세요. 
   - 사용 가능한 경우, 작업 전 `git add .` 와 `git commit -m "backup before modifications"`를 수행하세요.

2. **사전 계획 공유 (Plan & Review)**
   - 즉시 코드를 수정하지 마세요. 
   - 어떤 파일을 어떻게 수정할 것인지(수정/추가/삭제되는 함수나 CSS 클래스 등) 간략한 계획을 먼저 사용자에게 설명하고 동의를 구하세요.

3. **파괴적 변경 금지 (No Destructive Changes)**
   - 기존의 핵심 로직이나 구조를 완전히 삭제해야 하는 경우, 삭제 대신 주석 처리를 하고 `<!-- [DEPRECATED: YYYY-MM-DD] 사유 -->` 텍스트를 남기세요.
   - 확실하게 불필요해진 코드라도 사용자의 명시적인 '삭제 승인'이 있기 전까지는 삭제하지 마십시오.

4. **변경 이력 문서화 (Changelog)**
   - 수정이 완료되고 정상 작동이 확인되면, 프로젝트 루트의 `CHANGELOG.md` 파일에 오늘 날짜와 함께 변경된 사항(작업 내용, 수정된 파일 목록)을 한 줄 작성하여 업데이트하세요.

5. **점증적 수정 (Baby Steps)**
   - 한 번에 너무 많은 파일이나 기능을 동시에 수정하지 마세요. 하나의 기능을 고친 후 테스트/확인하고, 그 다음 작업으로 넘어가는 점진적 방식을 취하세요.

6. **수정 시 로컬 테스트 먼저, git push는 언제나 사용자가 수동으로 하도록**
   - 수정은 로컬 파일을 고치는 것 까지만 하고, 수정된 코드를 push 하는 것은 무조건 사용자가 수동으로 해야 합니다. 로컬 테스트 서버(localhost:3000)와 실제 URL(ophws.vercel.app)에서 둘 다 동작할 수 있는 방법으로 코드 수정을 제안해주세요.
